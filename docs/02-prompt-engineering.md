# 提示词工程对比分析

## Claude Code 系统提示词的核心发现

2026 年 3 月的源码泄露里，最有价值的不是代码本身，而是 `src/constants/` 里的系统提示词。
它展示了一个 $2.5B ARR 产品在生产环境里实际使用的提示词工程实践。

---

## 发现 1：提示词即协议

Claude Code 不写模糊指令，只写明确协议。

```markdown
# ❌ 模糊写法
请小心处理 git 操作，避免出错

# ✅ 协议写法
NEVER run force push to main/master
NEVER stage .env files or files containing credentials
```

**为什么有效：** LLM 对明确禁止的响应比对隐式期望稳定得多，尤其在复杂任务的中间步骤里。

**如何应用：** 审查你的 AGENTS.md，把所有「尽量」「建议」「注意」改成 NEVER/MUST/ALWAYS。

---

## 发现 2：边界条件显式化

Claude Code 的提示词对每一个边界情况都有明确处理：

```markdown
If a pre-commit hook fails...
If the file is too large...
If there are no changes to commit...
```

不写「处理好边界情况」，而是把每个边界情况都枚举出来。

**如何应用：** 对你最常出错的场景，逐一写出「如果 X 发生，那么做 Y」的规则，加入 AGENTS.md。

---

## 发现 3：静态/动态边界

Claude Code 的系统提示词有明确的边界标记，把不变的部分和每次刷新的部分分开。

```
[静态指令 — 永远不变]
...
[SYSTEM_PROMPT_DYNAMIC_BOUNDARY]
[动态上下文 — 每次刷新]
```

**为什么重要：** Prompt cache 只对完全相同的前缀有效。如果静态内容和动态内容混在一起，每次对话都无法命中缓存。

**量化价值：** 静态层约 6,000-8,000 字符 ≈ 1,500-2,000 tokens，每次对话节省这部分的计算费用。高频使用下积累的节省相当可观。

---

## 发现 4：上报路径明确化

Claude Code 定义了四种必须停下来问用户的情况：

1. 操作不可逆（删除、覆盖、force push）
2. 涉及外部发布（邮件、推文、GitHub commit）
3. 超出当前任务边界
4. 意图不明确

没有这个定义，agent 会用「我认为用户想要」来自行决策不可逆操作。

---

## 发现 5：反蒸馏机制（仅供了解，不需要移植）

Claude Code 在 API 请求里注入假工具定义（`anti_distillation: ['fake_tools']`），
目的是污染任何试图通过截取 API 流量来训练竞争模型的数据集。

这是 Anthropic 的商业防御机制，个人用户不需要也无法使用。

---

## 本方案 vs Claude Code 提示词质量对比

### 你的 SOUL.md（个性化维度）

Claude Code 的系统提示词里没有这层，它是任务导向的工具，
不需要定义「你是谁」——只需要定义「做什么」。

你的 SOUL.md 在个性化和语气控制上远超 Claude Code，这是个人助手场景独有的优势。

### AGENTS.md（行为协议维度）

整合前的差距主要在三个地方：

| 维度 | 整合前 | 整合后 |
|---|---|---|
| 工具调用失败协议 | 只有「重试前先回忆」 | 完整的失败处理路径 |
| 上报触发规则 | 仅「外部动作先确认」，边界模糊 | 5 条明确的触发场景 |
| 规则优先级 | 无，冲突时行为不可预测 | 4 层优先级，显式声明 |

---

## 最容易忽略的细节

**规则密度不等于规则质量。** Claude Code 的系统提示词并不长，
但每一条规则都是边界情况明确的操作性指令。

宁可写 10 条精确的规则，也不要写 50 条模糊的建议。
