# 设计思路详解

## 为什么要整合 Claude Code 的工程思路？

Claude Code 和 OpenClaw 是两个定位完全不同的工具：

- **Claude Code**：代码编辑 CLI，单一任务导向，上下文是项目仓库
- **OpenClaw**：多渠道个人助手，持久化运行，上下文是你的整个生活

但两者面对同一个底层问题：**如何让 LLM 在长期运行中保持稳定、高效、可预测的行为。**

Claude Code 有 $2.5B ARR 的商业压力推动工程优化，它踩过的坑（压缩失败、token 浪费、工具死循环）是真实生产环境里验证过的问题。2026 年 3 月的源码泄露让这些工程经验第一次可以被外部学习。

---

## 核心架构决策

### 决策 1：提示词分层

**问题：** 每次对话都把 SOUL.md、AGENTS.md、MEMORY.md 全部重新发送，静态内容重复计费。

**Claude Code 的解法：** `SYSTEM_PROMPT_DYNAMIC_BOUNDARY` 模式——静态内容固定在边界前，动态内容在边界后按需注入。

**本方案的实现：**

```
<!-- DYNAMIC_BOUNDARY --> 分隔符
```

插入在 HEARTBEAT.md 之后、MEMORY.md 之前。边界前的内容命中 prompt cache，边界后的内容每次刷新。

**收益：** SOUL.md + AGENTS.md + TOOLS.md 约 6,000-8,000 字符，稳定命中缓存，每次对话节省这部分的 token 费用。

---

### 决策 2：工具调用协议化

**问题：** 没有明确失败协议的 agent 在工具卡住时行为不可预测——可能无限重试、可能静默跳过、可能自动换工具绕过问题。

**Claude Code 的解法：** 系统提示词里对每种失败场景都有明确的处理路径，不依赖 LLM 自行判断。

**本方案的实现：** AGENTS.md 里的「工具调用失败协议」段落，定义了重试上限（2次）、失败后行为（停止+报告）、核心业务工具的特殊处理（立即告警不重试）。

---

### 决策 3：记忆三层架构

**问题：** autoCapture 只进不出，长期运行后记忆碎片堆积，检索质量下降。

**Claude Code 的解法：** autoDream 后台子 agent 定期整合记忆。

**本方案的实现：**

```
实时层：autoCapture（每轮最多3条原子记忆）
    ↓ 每周整理
周级层：memory-compaction（删除低价值、合并相似碎片）
    ↓ 每2-4周提炼
永久层：/evolve（高频模式 → AGENTS.md 规则）
```

三层对应三个不同的时间尺度，缺少任何一层都会导致记忆系统退化。

---

### 决策 4：规则语言标准化

**问题：** 模糊指令（「尽量」「建议」「注意」）导致 LLM 行为不可预测，在压力场景下更容易被忽略。

**Claude Code 的解法：** 系统提示词大量使用 NEVER/DO NOT 明确禁止，而不是软性建议。

**本方案的实现：** AGENTS.md 所有规则统一为 NEVER/MUST/ALWAYS 格式，并在文件开头声明优先级，解决规则冲突时的歧义。

---

## 与 ECC（Everything Claude Code）的关系

ECC 是针对 Claude Code 开发工作流的增强框架，和本方案解决的是不同问题：

| | 本方案 | ECC |
|---|---|---|
| 目标 | OpenClaw 个人助手优化 | Claude Code 开发工作流增强 |
| 核心价值 | 记忆系统 + 提示词架构 | Agents + Skills + 代码质量工具 |
| 直接借鉴 | /evolve 的设计思路 | 本方案参考了其 Continuous Learning v2 |

---

## 适配其他 AI Agent 框架

本方案的核心思路不依赖 OpenClaw 特有的功能，可以适配到其他框架：

**提示词分层：** 任何支持 system prompt 的框架都可以实现静态/动态分离，分隔符可以是任意约定的标记。

**工具调用协议：** 这是纯 prompt engineering，把「工具调用失败协议」段落加到任何 agent 的 system prompt 里都有效。

**记忆三层架构：** 需要替换 LanceDB 相关的具体实现，但删除低价值记忆 + 合并相似碎片的逻辑是通用的。

**规则语言标准化：** 纯 prompt engineering，不依赖任何框架。
